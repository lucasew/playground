# Repository specification

This is a draft of how a mediakit repo should look like.

A repo is basically a folder full of files. All the descriptions
of entities are made using text files. More structured data
can be represented as JSON.

These entities can carry assets such as a blog post can carry
the downloaded post or a YouTube video can carry the video itself.

## Architecture
The idea of this tool is to leverage faster machines to prepare
content to slower machines in a non-zucked up way.

Both machines communicate using any file syncronization system
like Git, syncthing or something else. The communication consists
on a shared folder that holds information about the sources
and the content itself so the user of the weaker device can
consume all the content without having Internet.

This tool can be as private as the storage medium and syncing
system used.

No encryption is done by this tool. The nearest thing from
encryption you will see here is the hashing of values that is
used to give names to files. The extractors can use encryption
but that specific situation is out of scope.

### Agents of this system
- **The worker machine**: faster one
- **The client machine**: slower one, used to consume the content itself
- **The syncing medium**: shared storage? syncthing? git? your choice!

## Entity types

At first, the following core entities are defined.

### Feeds
A feed is identified by it's source and has a set of posts.

One feed does not necessarily need to be a RSS feed, but RSS
feeds are the fallback extractor.

Each feed must have a canonical URL. Not the RSS feed itself,
for example, but the source itself. The final feed must be inferred
from the initial URL. Cache files that are not matched by any rule
can be created to help the extractors and will be maintained as
repo state.

### Posts
Posts can be assigned to feeds or defined by feed extractors and
represent one unit of content with a canonical URL. A post may have
more than one related data, like a article text, video, audio,
transcriptions and other resources that can be downloaded
or generated by extractors.

**Note about resources**: The post folder itself must only hold
stuff that helps extractors do it's job. Preferentially, no secrets.
All the assets themselves must be stored in the post cache in a way
that deleting them would not compromise the operation of the extractors.

### The post cache
The post cache holds data downloaded from posts such as audio, articles
and video. **There is no specific format for each kind of data yet**.

Items that belong here must not be downloaded eagerly.

### The post queue
Think of this like Pocket/Instapaper, but for anything else. This is
a folder where the posts are referenced so a extractor can fetch the post
cache following a policy such as the 50 more recent posts.

That way all the stuff can be fetched by one machine then replicated
using something like Syncthing on the local network to fresh up the
new stuff.

The client is responsible for submitting items to the post queue and, by
extension, to define the policy. A script that runs on the worker side
can also be used to add items to the queue.

The post queue is a folder with simple text files that hold the canonical
URL of the posts.

## The workflow

- After configuring the worker machine with the dependencies, the user
 creates a repository
- With the repository the user finds a way to sync the repository to the
 client device
- The user, either using the client or the worker, adds a feed, or a post.
- After syncing, if using the client, the user runs a command to start this
 program.
- The program kicks in and starts fetching data from feeds and data about
 posts.
- The user either sets up a auto-queue policy for some feed or add manually
 posts to the queue.
- The worker, after syncing, starts downloading the content itself, that
 after syncing is available in the client device.
- The user consumes the content. It's client must be able to mark the
 content as consumed and remove from the queue.
- The worker, after syncing, may delete the content and this deletion will
 be propagated at the next sync.

## Repo structure
- `mediakit.conf`: INI based configuration file for the extractors. Each
 extractor can use one or more configuration sections.
- `feeds`: Where the feed metadata is stored.
    - `:urlhash`: `sha256(url)` of the canonical URL of the feed. Must not be
     the RSS feed URL itself, for example.
        - `:unix-timestamp`: File that references a post URL. The name of the
         file is the unix timestamp with seconds precision of when the
         content is published.
        - `title`: Title of the feed.
        - `subtitle`: Subtitle of the feed.
        - `url`: The canonical URL of the feed. The hash of this file, that
         must not have a new line at the end, is used to name the folder.
        - `icon:*` Icon of the feed, different extractors can give more than
         one alternative icon.
        - `rss:feed_url`: (SUGGESTION) A RSS extractor can use a file with
         this name to store the resolved RSS feed URL.
        - `rss:last_updated`: (SUGGESTION) A RSS extractor can use a file
         with this name to store when was the last time the feed was updated.
- `posts`: Where the post data is stored.
    - `:urlhash`: `sha256(url)` of the canonical URL of the post. Must remove
     tracker query params such as `utm_*`.
        - `title`: Title of the post
        - `rss:*`: (SUGGESTION) Items provided by a RSS extractor.
        - Extractor data that is not necessarily the data itself can be
         stored here. Preferentially not more than a few kilobytes.
- `posts-cache`: Where the post assets are stored
    - `:urlhash`: `sha256(url)` of the canonical URL of the post. Must remove
     tracker query params such as `utm_*`.
        - `video:*`: Anything that starts with this will be treated as video
        - `video-264:*` Video extracted but converted to the h264 codec to be
         played by older devices. Shouldn't be with more quality than 720p.
        - `audio:*` Anything that starts with this will be treated as audio
        - `article:*` Markdown file with the article content. Must not use
         any non trivial Markdown feature. Links must be relative to the
         `article-assets` folder.
        - `article-assets`: Data referenced by the article markdown.
- `queue`: Folder with text files. The queue is sorted by the name of the
 files.
    - `:anything`: Text file that only holds the canonical (as hashed) link
     to one post.
- `extractors`: Folder with extra extractors as python modules. Doesn't have
 to have a __init__.py file.

## Extractors
Extractors are Python modules, that can abstract their behaviour with other
submodules, and are responsible to extract data for posts and feeds, or as a
driver to delegate to other tools such as yt-dlp and ffmpeg.

Extractors must not run stuff at the top level, so they should define a
function that the software
pulls and runs.

The software may define some extractors, such as a RSS feed parser.

Repo extractors must expose the following values:

- `EXTRACTOR_NAME`: used to identify the extractors on the logs.
- `setup`: runs at extractor initialization time. If a exception is catched
 there then the module is ignored.

The following functions, if returning None or nonexistent, are ignored.
- `get_feed_extractor`: receives a url string and returns a function that
 receives the feed directory as `pathlib.Path`.
- `get_post_extractor`: receives a url string and returns a function that
 receives the post directory as `pathlib.Path`.
- `get_post_cache`: receives a url string and returns a function that
 receives the post directory and the post cache directory as `pathlib.Path`.

The `setup` function and the `get_*` run in sort order based on
`EXTRACTOR_NAME`. A extractor that gets the data downloaded from a video and
converts it to h264 preferencially should be named with a few `z` at the
start of the name.
